Redis 数据结构

Redis 支持五种数据类型：

  • 字符串（String）：字符串是 Redis 最基本的数据类型，可以用来存储任意类型的数据。
  • 哈希（Hash）：哈希是一种键值对的集合，可以用来存储对象或记录。
  • 列表（List）：列表是一种有序的字符串列表，可以用来存储数据序列。
  • 集合（Set）：集合是一种无序的字符串集合，可以用来存储唯一的元素。
  • 有序集合（Sorted Set）：有序集合是一种有序的字符串集合，每个元素还带有一个分值，可以用来存储有序的数据。

Redis 的数据结构都使用了不同的数据结构来实现：

  • 字符串：字符串使用简单动态字符串（SDS）来实现。SDS 是一种动态扩展的字符串，可以根据需要自动扩展。
  • 哈希：哈希使用哈希表来实现。哈希表是一种快速查找的数据结构，可以用来快速查找特定的键值对。
  • 列表：列表使用双向链表来实现。双向链表是一种可以从任意位置进行插入或删除元素的数据结构，因此非常适合列表的数据结构。
  • 集合：集合使用整数数组来实现。整数数组是一种可以快速查找元素是否存在的数据结构，因此非常适合集合的数据结构。
  • 有序集合：有序集合使用跳表来实现。跳表是一种可以快速查找元素并按分值排序的数据结构，因此非常适合有序集合的数据结构。

Redis 的数据结构设计非常巧妙，既保证了数据的存储效率，又满足了不同场景的需求。

Redis 持久化

Redis 是基于内存的 NoSQL 数据库，但它也支持持久化，可以将内存中的数据保存到磁盘上。Redis 的持久化有两种方式：RDB 和 AOF。

RDB 持久化

RDB 持久化是 Redis 默认的持久化方式。RDB 持久化是周期性的将内存中的数据快照保存到磁盘上。Redis 的 RDB 持久化原理是将内存中的数据全部序列化成一个二进制文件，然后将该文件保存到磁盘上。

优点

  • 简单易用，配置简单
  • 性能较高，持久化过程不会影响数据的读写

缺点

  • 数据不完整，RDB 持久化只能保存内存中的数据快照，因此如果服务器在持久化过程中发生异常，可能会导致数据丢失。
  • 不支持增量持久化，RDB 持久化需要将内存中所有数据都保存到磁盘上，因此对于数据变化频繁的场景，持久化时间会比较长。

AOF 持久化

AOF 持久化是将所有对数据库的写操作记录到一个日志文件中，然后在服务器重启时，将日志文件中记录的所有操作重放一遍，以此来恢复数据。

优点

  • 数据完整，AOF 持久化记录了所有对数据库的写操作，因此即使服务器在持久化过程中发生异常，也能保证数据的完整性。
  • 支持增量持久化，AOF 持久化只需要记录对数据库的新增或修改操作，因此对于数据变化频繁的场景，持久化时间会比较短。

缺点

  • 性能较低，AOF 持久化需要记录所有对数据库的写操作，因此会增加服务器的写负载。
  • 日志文件可能非常大，AOF 持久化会产生一个日志文件，如果数据变化频繁，日志文件可能会非常大，因此需要定期清理日志文件。
