Spring入门

对Spring的理解：

Spring框架 划分多个模块.。应用程序去选择他们需要的模块。

core是核心容器的模块，包含一个配置模型、一个依赖注入机制。

Spring框架还是为蛮多应用架构提供基础支持，：消息传递、事务性数据、持久性以及WEB。还有包含Servlet的SpringMVC框架，以及并行的Spring WebFlux响应式Web框架。

对Spring框架遵循的原则

  • 在每个层面提供选择。
  • 适应不同的观点
  • 保持强大的后向兼容性
  • 关心API的设计
  • 为代码质量设定高标准 。（强调有意义、最新和准确的javadoc）

Spring核心技术

最重要：Spring框架的反转控制（IoC）容器。在对Spring容器彻底处理后，紧接着是对Spring面向切面编程（AOP）

Spring框架有自己的AOP框架

目前Java企业中功能最丰富、最成熟的AOP实现： AspectJ

AOT处理可以提前优化你的应用程序。通常用于使用GraalVM的原生镜像部署

IoC容器理论：

IoC是Spring框架的核心思想； 优点在于解耦

为什么说IoC会有解耦的优点呢？

原因在于：在此前，满足一些需求时，一般都在service层去实例化一个dao层对象，来实现参数/数据传递（此时耦合度增加）

但是我们在项目的原则：高内聚低耦合，因此才会出现IoC容器来实现。

那IoC容器又是怎么去实现解耦的？

实现：Spring将需求实现的控制权（其实是实例化对象的控制权）给用户，将实例化过程给IoC容器，通过IoC去和service层联系，来实现需求。（也达到了：降低耦合度，更加灵活）

总结：控制反转是通过描述（xml或者注解），并通过第三方（IoC）生产或者获取特定对象的方式。

  • 在Spring中实现控制反转：IoC容器
  • 实现方式：依赖注入（DI ） — 本质：使用set方法实现
  • 底层：反射机制，通过全类名去获取属性和方法

注意：这里的IoC和依赖注入不一样，DI只是IoC的一种实现方法，因为IoC是一种思想。

首先先了解Bean是什么？

本质上是指代任何被Spring加载生成出来的对象。（区别于JavaBean，因为JavaBean是对于Java类的一种规范定义。）

Spring Bean 代表着Spring中最小的执行单位，其加载、作用域、生命周期的管理都是由Spring操作。

下面是列举的一些Bean含义：

  • Java面向对象、对象有方法和属性、那么就需要对象实例来调用方法和属性（说白了就是“实例化”）
  • 凡是有方法或者属性的类都需要实例化，这样才能具象化去使用这些方法和属性
  • 规律：凡是子类及带有方法或者属性的类都要加上注册Bean到Spring IoC的注解
  • Bean相当于类的代理（实际上通过反射和动态代理来实现的）
  • 我们都在微博上@过某某，对方会优先看到这条信息，并给你反馈，那么在Spring中，你标识一个@符号，那么Spring就会来看看，并且从这里拿到一个Bean（注册）或者给出一个Bean（使用）

AOP总结

概念：

  • AOP面向切面编程，一种编程范式

作用

  • 在不改动原来设计的基础上为方法进行功能增强

核心概念

  • 代理（Proxy：SpringAOP的核心本质：代理模式实现
  • 连接点（JoinPoint（jp）：可以理解为任意方法的执行
  • 切入点（Pointcut：匹配连接点的表达式，具有共性功能的方法描述
    • 获取切入点参数
      • jp：适用于除环绕通知之外的所有，设置为方法的第一个形参
      • pjp：适用于环绕通知
  • 通知（Advice：多个方法的共性功能，在切入点出执行
    • 环绕通知（@Around）：
      • 依赖形参ProceedingJoinPoint（pjp）才能实现对原始方法的调用
      • 可以隔离原始方法的调用执行
      • 返回值设置为Object类型
      • 可以对原始方法中出现的异常进行处理
  • 切面（Aspect：描述通知与切入点的对应关系
  • 目标对象（Target：被代理的原始对象变成目标对象。

针对于框架的学习的知识点

MyBatis的一级缓存和二级缓存的区别：

前言：一级缓存是默认开启的，二级缓存是手动配置开启的

一级缓存：

  • 默认开启
  • 底层是基于HashMap实现的本地缓存
  • 作用域是session
  • 不同的sqlSession之间互不影响

二级缓存：

  • 需要手动配置开启
  • 属于Mapper级别的缓存
  • 在同一个命名空间的所有操作，都会改变缓存的数据

所以建议：使用第三方插件来当二级缓存，如redis等。

spring aop理解

aop概念：

  • 面向切面编程，它是oop（面向对象编程）的补充；

  • 因为在oop中是以类为基本单元，在aop中是以切面为基本单元

对于面向切面编程的理解：

  • 主要就核心业务功能和周边功能是分别独立进行开发，二者不是耦合的；
  • 此时把切面功能和核心业务功能整合在一起就是aop

aop的目的：

  • 对程序解耦，减少系统的重复代码，提高拓展性和可维护性

注意不同Spring版本之间的aop通知顺序是不一样的，Spring5.28 为分界线

aop的实现：

  • 关键在于aop框架自动创建的aop代理，分为静态代理、动态代理
  • 静态代理：AspectJ；是编译期实现
  • 动态代理：SpringAop；是运行期实现

下面我们就来讲讲：什么Aop的两种代理实现机制？

首先：Aop动态代理实现机制有两种：jdk动态代理和CGLIB动态代理

JDK：

  • 实现接口，利用java反射完成对类的动态创建

  • 核心是InvocationHandler接口、Proxy类

CGLIB：

  • 继承目标对象，然后操作子类的字节码生成代理类
  • 注意：被继承的类不能是final修饰，否则无法使用CGLIB的动态代理的对象

什么是Spring的事务

事务的概念：

  • 事务其实是一个并发控制单位，是用户定义的一个操作序列，操作要么全部完成，要么全部不完成，是一个不可割舍的单位。
  • 有ACID四个特性：原子性、一致性、事务隔离、持久性。
  • 支持两种方式：
    • 声明式事务（注解方式，使用多
    • 编程式事务（代码方式

Spring事务本质：就是通过SpringAOP切面技术，在合适的地方开启事务，接着在合适的地方提交或者回滚事务，

从而实现业务编程层面的事务操作

事务传播类型：

  • 有七种传播类型：REQUIRED、SUPPORTS、MANDATORY、REQUIRES_NEW、NOT_SUPPORTED、NEVER、NESTED

  • 常用的有三：

    • REQUIRED（required）：默认的，父子方法共同使用一个事务

    • REQUIRES_NEW（requires_new)：父子事务相互独立，并不会互相影响

    • NESTED（nested）：

什么时候Spring事务会失效？

  • 当在同一个类中，其他没有@Transactional注解的方法中调用有@Transactional注解的方法，有该注解的方法失效。
  • 如果直接调用，不通过对象调用，也会失效。因为Spring事务是通过AOP实现的。
  • @Transactional注解只能作用在public方法上。
  • 被@Transactional注解的方法所在类：必须被Spring管理。

logging是Spring中唯一强制的外部依赖，Spring中默认使用的日志是commons-logging，简称：JCL，这里所说的强制性，是因为在Spring-core这个模块中引入该依赖。

Spring中Bean是线程安全的吗？怎么解决该问题？

先说结论，再说解决方式

结论：

​ Spring中的Bean不是线程安全的。

原因：

​ 因为Spring容器本身并没有提供Bean的线程安全策略，因此可以说Spring容器不具备线程安全的特性。

解决：

  • 首先：
    • 对于prototype作用域，每次生成一个新的对象，所以不存在线程安全问题。
    • 对于Singleton作用域，默认是线程不安全的，但是对于开发中大部分的Bean，其实都是无状态，不需要保证线程安全。
      • 无状态：这个实例没有属性对象，不能存实数据，比如controller、service、dao
      • 有状态：实例有属性，可以保存数据，线程不安全，比如：pojo
  • 方法：
    • 可以将Bean的作用域改为prototype
    • 可以采用ThreadLocal来解决线程安全，因为会为每一个线程保存一个副本变量，每一个线程只操作自己的副本变量

SpringMVC的工作原理

什么是是SpringMVC？

  • 是基于Java实现MVC的轻量级web框架
  • 是基于请求驱动的web框架，使用前端控制器（DispatcherServlet）模式设计的

SpringMVC优点

  • 轻量级，简单易学
  • 高效，基于请求响应的MVC框架
  • 约定优于配置
  • 与Spring无缝结合

Spring MVC工作原理：

SpringMVC工作的组件：
  • 前端控制器：接收请求，响应结果
  • 处理器映射器（HandlerMapping）：根据url查找处理器
  • 处理器（handler）：程序员编写代码处理逻辑
  • 处理器适配器（handlerAdapter）：将处理器包装成适配器
  • 视图解析器（ViewResovler）：进行视图解析，多返回的字符串进行处理，可以解析成对应的页面。
流程：
  • 1：用户发起请求到DispatchServlet
  • 2：dis请求（handlerMappering）处理器映射器去找相应的处理器（handler）：通过xml配置或者注解查找
  • 3：找到以后handlerMappering向dis返回“执行链”
  • 4：dis调用handlerAdapter（处理器适配器）去执行处理器（handler）
  • 5：handlerAdapter执行handler
  • 6、handler执行完，给handlerAdapter返回modelAndView
  • 7：handlerAdapter向dis返回modelAndView（层层递进）
  • 8：dis请求视图解析器（viewResolver）去进行视图解析
  • 9：视图解析器向dis返回view
  • 10：dis对视图进行渲染
  • 11：dis向用户响应结果

SpringBoot的自动配置

Spring Boot启动的时候：

  • 会通过**@EnableAutoConfiguration注解找到META-INF/spring.factories配置文件中的所有自动配置类，并对其进行加载，这些自动配置类都是以AutoConfiguration结尾**来命名的，
  • 它实际上就是一个JavaConfig形式的Spring容器配置类，通过@Bean导入到Spring容器中，**以Properties结尾命名的类**是和配置文件进行绑定的。
  • 它能通过这些以Properties结尾命名的类中取得在全局配置文件中配置的属性，我们可以通过修改配置文件对应的属性来修改自动配置的默认值，来完成自定义配置

详细了解请看：Spring Boot面试必问：自动配置原理_spring boot自动配置原理-CSDN博客

SpringBoot的核心注解有哪些？他主要是由那几个注解组成？

核心注解：@SpringBootApplication：

  • 核心注解是@SpringBootApplication，它主要由@SpringBootConfiguration，@EnableAutoConfiguration和@ComponentScan这三个构成。

@SpringBootConfiguration：

  • 里面就只有一个@Configuration主要注解，也就是把该类变成一个配置类。
  • 所以@SpringBootConfiguration就相当于@Configuration

@EnableAutoConfiguration：

  • 是由 @AutoConfigurationPackage：是自动配置包，里面包含@Import注解:用来给容器导入了自动配置包的注册器
  • 和@Import(EnableAutoConfigurationImportSelector.class)：导入自动配置导入选择器组件,

@ComponentScan：

  • 扫描包，该注解默认会扫描该类所在的包下所有的配置类

cookie、session、jwt的区别

cookie是什么？

  • cookie存储在客户端（浏览器本地缓存）
    • 会在浏览器下次向同一服务器再次发起请求时携带并发送到服务器上
  • cookie不可以跨域
    • 一级域名和二级域名之间是允许共享使用的（靠的是domain）

session是什么？

  • session是一种记录服务器和客户端会话状态的机制
  • 是基于cookie实现的，session存储在服务器端，
  • sessionID存储在客户端的cookie中

联系：sessionID是链接cookie和session的节点。

cookie和session的区别：

  • 安全性：session比cookie安全。
  • 存取值的类型不同：cookie只支持String类型（其他类型强转String）；session可以是任意类型
  • 有效期不同：cookie可以长时间保存；session一般失效时间短，（客户端关闭或者session超时）
  • 存储大小不同：单个cookie最多存储4k；session可存储远超cookie。但是当访问量大，会占用过多服务器资源

令牌：

  • 特点：
    • 服务端无状态化，可扩展性好
    • 支持移动端设备
    • 安全
    • 支持跨程序调用

什么是JWT？

  • 是为了在网络应用环境间传递声明而执行的一种基于 JSON 的开放标准（RFC 7519）
  • JSON Web Token（简称 JWT）是目前最流行的跨域认证解决方案。
  • 是一种认证授权机制。

使用方式：

  • 方式一：客户端收到服务器返回的 JWT，可以储存在 Cookie 里面，也可以储存在 localStorage

  • 方式二：跨域的时候，可以把 JWT 放在 POST 请求的数据体里。

  • 方式三：通过 URL 传输

token和jwt的异同：

此处注意哈：token和jwt是不一样是，下面我们就说说他们的异同：

相同：

  • 都是访问资源的令牌
  • 都是可以记录用户信息
  • 都是服务端无状态化
  • 都是只有认证成功后，客户端才能访问服务端受保护的资源

不同：

  • token：服务端验证客户端发来的token，同时还需要查询数据库获取用户信息，然后验证token是否有效

  • JWT：是将token和payload加密后存储于客户端，服务端只需要密钥的校验即可，减少访问查询数据库，

    ​ 因为jwt包含用户信息和加密数据。

如果还想详细了解的话，可以查看：还分不清 Cookie、Session、Token、JWT？ - 知乎 (zhihu.com)

过滤器和拦截器的区别

过滤器概念：

  • 是在java web中将你传入的request、response提前过滤掉一些信息，或者提前设置一些参数。
  • 然后再传入Servlet或Struts2的 action进行业务逻辑处理

拦截器概念：

  • 是面向切面编程（AOP，Aspect Oriented Program）的。
  • 就是在你的Service或者一个方法前调用一个方法，或者在方法后调用一个方法

区别：

拦截器：

  • 基于Java的反射机制
  • 不依赖与Servlet容器
  • 只对action请求起作用
  • 可以访问action上下文、值、栈对象
  • 在action的生命周期中：可以多次调用
  • 可以获取ioc容器中的所有Bean

过滤器：

  • 基于函数的回调
  • 依赖于Servlet容器
  • 可以对几乎所有的请求起作用
  • 不可以
  • 只能在容器初始化调用一次
  • 不可以

运行SpringBoot有哪几种方式：

  • 打包用命令或者放到容器中运行
  • 用Maven插件运行
  • Gradle插件运行
  • 直接执行main方法执行（启动类）

@RequestMapping注解中的属性consumes和produces，分别和请求头中的信息

  • 属性produces：指定返回值类型，并且可以设置返回值类型和返回值的字符编码；

  • 属性consumes： 指定处理请求当中的提交内容类型（Content-Type）：application/json, text/html等;

  • handlerAdapter
